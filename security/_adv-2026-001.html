ADV-2026-001 CLEANSTER ● DISCLOSED — no vendor response received
A black-box assessment of Cleanster's public web application and API surface identified eight findings, ranging from plaintext password storage and email/link injection in the password-reset flow to an exposed production backend, an unrestricted Google Maps API key, and a decodable session JWT. All findings were confirmed on a dedicated test account; no destructive testing was performed against real user data.

01  Plaintext Password Exposure via Password Reset CRITICAL · 7.5
Plaintext Storage of a Password · CWE-256 · PLNK-2026-001  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N
DESCRIPTION
Submitting a request to POST /forgotpincode with a registered account's email causes the server to email that account's actual current password in plaintext, rather than issuing a time-limited reset token. This was first noticed when a legitimate password-reset request returned the account's real, still-valid password in the email body, and was independently reproduced against a dedicated test account. This confirms passwords are stored in a reversible or unhashed format on the backend, rather than a proper salted hash.
IMPACT
Anyone with access to a victim's email inbox — via a compromised mail provider, shared device, or intercepted message — obtains their live Cleanster password in plaintext, with no need to crack or reset anything. It also means any future or historical breach of Cleanster's database would expose every user's real password directly, since none of them appear to be hashed. CVSS reflects a High technical score, but the real-world severity is treated as Critical here given the systemic nature of the underlying storage flaw.
REMEDIATION
Store passwords with a modern salted hash (bcrypt, scrypt, or Argon2) and remove any backend capability to retrieve the original password. Replace the plaintext-password-email flow with a single-use, time-limited reset link.
02  Unsanitized Parameter / Email Body Injection HIGH · 5.3
Improper Neutralization of Special Elements (Injection) · CWE-74 · PLNK-2026-002  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:L/A:N
DESCRIPTION
The ipaddress field in the JSON body of POST /forgotpincode is reflected directly, unsanitized, into the body of the outbound password-reset email. Sending "ipaddress": null produced an email addressed to "user null"; setting the value to an arbitrary string reproduced that string verbatim in the email. A CRLF injection attempt (%0d%0a) to manipulate mail headers was tested and did not succeed — special characters were rendered literally rather than interpreted.
IMPACT
An attacker can inject arbitrary attacker-controlled text into a legitimate, authenticated Cleanster email sent to any account holder, without needing credentials. While header injection was not achieved, body content manipulation alone is sufficient to alter what a trusted-sender email appears to say.
REMEDIATION
Sanitize and/or strictly type-validate all request parameters before interpolating them into outbound email templates. Treat any client-supplied value used in an email as untrusted output and encode or reject it accordingly.
03  Link Injection Enabling Trusted-Sender Phishing HIGH · 4.7
URL Redirection to Untrusted Site ('Open Redirect'-adjacent content injection) · CWE-601 · PLNK-2026-003  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:R/S:C/C:N/I:L/A:N
DESCRIPTION
Building on Finding 02, the same ipaddress parameter on POST /forgotpincode was found to render injected URL strings as clickable hyperlinks in the delivered email. An attacker-supplied URL placed in this field appears as a live link in an otherwise entirely legitimate Cleanster email.
IMPACT
An attacker can embed a malicious, attacker-controlled link inside a real password-reset email sent from Cleanster's own infrastructure. Because the sender, formatting, and surrounding content are all genuine, this is a highly effective phishing vector — the victim has no reliable way to distinguish it from a legitimate reset email, and typical "check the sender" phishing advice does not protect against it.
REMEDIATION
Apply the same input sanitization recommended in Finding 02, and additionally strip or escape any URL-like patterns from user-controllable fields before they reach an email template. Consider rendering user-influenced fields as plaintext only, never as HTML.
04  User Enumeration via Password Reset LOW · 5.3
Observable Discrepancy · CWE-203 · PLNK-2026-004  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N
DESCRIPTION
POST /forgotpincode returns a different server response depending on whether the submitted email address corresponds to a registered Cleanster account. Testing with a known-registered address and a known-unregistered address consistently produced distinguishable responses.
IMPACT
An attacker can silently confirm whether any given email address has a Cleanster account, without triggering any visible notification to that user. This enables targeted attacks (credential stuffing, phishing, or abuse of Finding 01) against confirmed-valid accounts, and at scale, could be used to harvest lists of valid Cleanster users.
REMEDIATION
Return an identical, generic response regardless of whether the submitted email is registered (e.g., "If an account exists for this email, a reset link has been sent"). Ensure response timing is also normalized to prevent timing-based enumeration.
05  Exposed Production Backend URL MEDIUM · 5.3
Exposure of Sensitive Information to an Unauthorized Actor · CWE-200 · PLNK-2026-005  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N
DESCRIPTION
Passive traffic analysis of api.cleanster.com revealed requests resolving to a Google App Engine URL in the form [app]-prod-dot-official-[name]-project.ue.r.appspot.com. Separately, requesting api.cleanster.com/robots.txt issues a redirect directly to this full production backend URL, exposing it to any unauthenticated visitor.
IMPACT
Exposing the real backend hostname lets an attacker map Cleanster's underlying infrastructure and target scanning, abuse, or attacks directly at the App Engine origin, potentially bypassing any protections (CDN, WAF, rate limiting) that sit in front of the public-facing domain.
REMEDIATION
Remove the redirect that discloses the raw App Engine hostname, and ensure any front-end proxy or CDN fully masks the backend origin. Restrict direct access to the App Engine URL, or place equivalent protections directly on it.
06  Unrestricted Google Maps API Key MEDIUM · 5.3
Use of Hard-coded Credentials · CWE-798 · PLNK-2026-006  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:L
DESCRIPTION
A Google Maps API key was found hardcoded in Cleanster's client-side page source / JS bundle. The key was confirmed active and unrestricted by calling Google's geocoding endpoint directly with it and receiving a valid geocoding response.
IMPACT
Because the key is unrestricted (no domain/IP/API restriction enforced), anyone who extracts it from the page source can issue requests against it directly, outside of Cleanster's own application. At volume, this can generate substantial unexpected billing on Cleanster's Google Cloud account, or exhaust API quota and disrupt the map features of the legitimate app.
REMEDIATION
Restrict the API key by HTTP referrer and/or API scope in the Google Cloud Console, rotate the current key immediately, and set a billing alert/quota cap as defense in depth.
07  Exposed Session JWT (Informational) LOW · 3.7
Insufficiently Protected Credentials · CWE-522 · PLNK-2026-007  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N
DESCRIPTION
A session JWT was observed in Burp Suite's HTTP history during normal authenticated browsing. Decoding it at jwt.io confirmed it contains user session data. An "alg: none" signature-bypass attack was attempted against it and did not succeed — the server correctly validated the token's signature.
IMPACT
This finding is primarily informational: the token's structure and claim contents are visible to anyone who can observe traffic (e.g., on a shared network without TLS interception protections), but the attempted forgery exploit was unsuccessful, meaning the server's validation appears sound. Listed for completeness and as a reminder that session tokens still warrant careful handling.
REMEDIATION
No urgent action required given failed exploitation; as good practice, ensure JWTs carry only the minimum necessary claims, use short expiries, and continue enforcing strict signature validation.
08  Missing Rate Limiting on Password Reset LOW · 4.3
Improper Restriction of Excessive Authentication Attempts · CWE-307 · PLNK-2026-008  ·  CVSS 3.1 AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:L
DESCRIPTION
No rate limiting was observed on POST /forgotpincode during testing. This was identified as a gap based on endpoint behavior rather than exhaustively load-tested, and is reported with that caveat.
IMPACT
Absent rate limiting, an attacker could repeatedly call the endpoint to flood a target user's inbox with password-reset emails — each one containing their live plaintext password per Finding 01 — or to enumerate accounts at scale using Finding 04. This finding meaningfully amplifies the severity of Findings 01 and 04.
REMEDIATION
Apply per-IP and per-account rate limiting to the password-reset endpoint, with exponential backoff and/or CAPTCHA after a small number of requests in a short window.
